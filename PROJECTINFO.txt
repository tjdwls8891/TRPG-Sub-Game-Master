1. 전역 클래스 및 데이터 모델 (Classes & Data Models)
    1.1. TRPGSession (in core.py)
        - 작동 방법: 단일 게임 세션의 모든 진행 상태, 채널 메타데이터, 플레이어/NPC 정보, 자원, 로그 배열, 캐시 메타데이터(생성 시점, 캐시 노트) 등을 담는 거대한 컨테이너 객체입니다. main.py의 bot.active_sessions 딕셔너리에 채널 ID를 키로 하여 적재됩니다.
        - 개발 이유 및 알고리즘 의도:
            * 상태 중앙화 및 파편화 방지: 비동기 이벤트(명령어, 채팅, 주사위 등)가 동시다발적으로 발생할 때, 매번 데이터베이스나 파일에서 상태를 개별적으로 읽어오는 것은 비효율적입니다. 세션과 관련된 모든 변수를 하나의 클래스 내부에 캡슐화하여, 어떤 모듈에서든 session = bot.active_sessions.get(channel.id) 단 한 줄로 전체 컨텍스트에 접근할 수 있도록 설계되었습니다.
            * 직렬화(Serialization) 최적화: 이 객체의 구조는 data.json의 스키마와 1:1로 대응하도록 짜여 있습니다. 따라서 메모리 상태를 즉시 디스크로 백업하거나(save_session_data), 디스크에서 메모리로 복구(restore_sessions_from_disk)하는 과정이 매우 직관적이고 무손실로 이루어집니다.

    1.2. PromptBuilder (in core.py)
        - 작동 방법: 턴 묘사를 생성할 때, 현재 세션의 상태를 바탕으로 AI에게 전달할 프롬프트 텍스트를 블록 단위로 조립합니다. add_memory_block, add_player_block 등의 메서드를 체이닝하여 순차적으로 텍스트를 누적한 뒤 build()로 최종 반환합니다.
        - 개발 이유 및 알고리즘 의도:
            * 할루시네이션(환각) 통제: LLM은 주어지는 프롬프트의 순서와 포맷에 큰 영향을 받습니다. 빌더(Builder) 패턴을 도입하여 '이전 요약 -> 플레이어 정보 -> 개입 중인 NPC -> 연관 기억 -> 최근 행동 -> 최종 룰 강제'라는 엄격한 정보 주입 순서를 시스템 레벨에서 강제합니다.
            * 동적 토큰 최적화 (트리거 시스템): add_triggered_npc_block과 add_keyword_memory_block의 알고리즘은 전체 데이터를 무지성으로 프롬프트에 넣지 않습니다. 객체 초기화 시 최근 10개의 로그와 GM 지시사항을 recent_logs_combined라는 하나의 문자열로 묶은 뒤, 특정 NPC의 이름이나 keyword가 이 문자열 안에 존재하는지(if kw in combined) 검사합니다. 스캔에 적중한 정보만 프롬프트에 주입하여, 토큰 낭비를 막고 불필요한 설정이 개입하여 묘사가 산으로 가는 현상을 방지합니다.

    1.3. TRPGBot (in main.py)
        - 작동 방법: discord.ext.commands.Bot을 상속받은 메인 객체로, 디스코드 API와의 통신을 유지하면서 active_sessions, session_io_locks, genai_client 등 전역 리소스를 내부 속성으로 쥐고 있습니다.
        - 개발 이유 및 알고리즘 의도: 전역 변수(Global Variable)의 무분별한 사용을 막기 위함입니다. 봇 인스턴스 자체에 API 클라이언트와 세션 딕셔너리를 종속시킴으로써, cogs 내부의 어떤 명령어에서든 self.bot을 통해 안전하게 상위 리소스에 접근하고 동시성 제어(asyncio.Lock)를 수행할 수 있게 합니다.

2. 핵심 유틸리티 함수 (Core Utility Functions)
    2.1. calculate_upload_cost & calculate_storage_cost & process_cache_deletion
        - 작동 방법: PRICING_1M 상수에 정의된 모델별 단가표(100만 토큰 기준)를 참조하여, 업로드 비용과 보관 비용을 분리 산출하고 1$=1500원의 고정 환율을 적용해 원화(KRW)로 반환합니다. 보관 비용은 캐시가 유지된 시간(초)을 분 단위로 반올림(15분 45초 -> 16분)하여 정밀하게 정산합니다. 파일에는 부동소수점 원형 그대로 기록되며, 알림 출력 시에만 소수점 셋째 자리에서 반올림(format_cost)됩니다.
        - 개발 이유 및 알고리즘 의도: 장기간 진행되는 TRPG 세션 특성상 API 과금액이 급증할 위험이 있습니다. 입력, 출력 토큰 외에도 Context Caching의 유지 비용과 캐시 읽기 할인율을 분리하여 계산함으로써, 프로젝트의 경제성을 실시간으로 모니터링할 수 있는 정밀한 과금 추적 시스템을 구축하기 위함입니다.

    2.2. build_scenario_cache_text(bot, model_id, scenario_data)
        - 작동 방법: 시나리오 JSON에서 세계관, 룰북, NPC 사전 등을 추출해 하나의 텍스트(rulebook_text)로 합칩니다. 이때 session.cache_note에 누적된 내용이 있다면 룰북 하단에 지연 병합(Lazy Merge)합니다. 이후 Gemini API로 토큰 수를 계산하고, 32,768 토큰 미만일 경우 . 기호를 곱하여 임의의 패딩(Padding) 영역을 덧붙인 뒤 반환합니다.
        - 개발 이유 및 알고리즘 의도: Gemini 모델의 Context Caching 시스템은 최소 32,768 토큰 이상일 때만 활성화되는 시스템적 한계가 있습니다. 소규모 시나리오 데이터라도 캐싱의 이점(비용 절감, 추론 속도 향상, 지시사항 망각 방지)을 누리기 위해, AI가 무시하도록 지시한 더미 데이터([System Data Padding Area])를 강제로 주입하여 최소 요구치를 우회 충족시키는 기술적 해킹(Hack)이자 최적화 알고리즘입니다.

    2.3. build_compression_prompt(session, log_text)
        - 작동 방법: 턴이 일정 수 이상 누적되었을 때, 이를 무손실에 가깝게 요약하기 위한 프롬프트를 조립합니다. 문학적 묘사를 배제하고 행동, 아이템 수치, 턴 발생 시점을 개조식으로 변환하도록 지시문을 구성합니다.
        - 개발 이유 및 알고리즘 의도: 아무리 컨텍스트 윈도우가 크더라도 대화가 수백 턴 누적되면 모델은 과거의 디테일(획득한 아이템, 과거에 다친 부위 등)을 망각하는 할루시네이션을 일으킵니다. 이를 방지하고자, 미사여구를 걷어내고 '인과율'과 '데이터(수치/상태)'만을 극도로 건조하게 압축하는 알고리즘을 도입했습니다. 이는 장기 세션의 기억을 보존하는 '무손실 압축 메모리 시스템'의 핵심입니다.

    2.4. save_session_data(bot, session)
        - 작동 방법: 메모리에 있는 TRPGSession 객체의 상태를 직렬화하여 해당 세션의 data.json 파일에 덮어씁니다. asyncio.to_thread를 사용하여 논블로킹(Non-blocking)으로 파일 I/O를 처리하며, session_io_locks를 통해 동시 접근을 제어합니다.
        - 개발 이유 및 알고리즘 의도: 비동기 환경(디스코드 봇)에서는 두 사용자가 동시에 명령어를 입력할 경우 레이스 컨디션(Race Condition)이 발생하여 파일이 깨질 위험이 있습니다. asyncio.Lock을 채널별로 할당하여 데이터 오염을 완벽히 차단하고, 봇이 예상치 못하게 다운되더라도 1턴 단위의 손실도 없이 즉시 복구할 수 있는 무결성을 보장합니다.

    2.5. restore_sessions_from_disk(bot)
        - 작동 방법: main.py의 on_ready (부팅) 시점에 실행됩니다. sessions 폴더를 순회하며 data.json을 읽어 TRPGSession 객체를 재조립합니다. 만약 저장된 캐시 이름(cache_name)을 통해 Google 서버에 접근했을 때 만료 에러(APIError)가 반환되면, 즉시 build_scenario_cache_text를 호출하여 새 캐시를 발급받아 연결합니다.
        - 개발 이유 및 알고리즘 의도: 세션의 영속성을 위한 로직입니다. 특히, 캐시 서버는 6시간(TTL)이 지나면 데이터를 삭제하므로, 봇이 재시작될 때 기존 캐시의 생존 여부를 핑(Ping) 테스트하고, 죽어있다면 조용히 재발급하는 자가 치유(Self-healing) 알고리즘을 구현하여 사용자의 게임 경험이 중단되지 않도록 설계했습니다.

    2.6. generate_character_details(bot, scenario_data, char_type, char_name, instruction, session_id)
        - 작동 방법: 캐릭터 설정 생성을 위해 프롬프트를 구성하고 Gemini LOGIC_MODEL에 요청을 보냅니다. char_type이 'pc'냐 'npc'냐에 따라 필수 기입 항목(성격, 소속 등)을 다르게 포맷팅하여 전달합니다.
        - 개발 이유 및 알고리즘 의도: GM의 짧은 지시만으로도 시나리오 세계관과 충돌하지 않는 정교한 설정 초안을 얻기 위함입니다. 일반 묘사용 모델 대신 추론 성능이 더 높은 LOGIC_MODEL (Pro 모델)을 지정하여 포맷 준수율을 높이고, 서술어를 뺀 개조식 단문 출력을 강제함으로써 생성된 텍스트가 이후 프롬프트에 재주입될 때 토큰 효율을 극대화하도록 설계했습니다.

    2.7. stream_text_to_channel(bot, channel, text, words_per_tick, tick_interval)
        - 작동 방법: 전체 문자열을 \n\n 단위로 쪼갠 후, 다시 띄어쓰기 기준으로 words_per_tick(예: 5단어)만큼 잘라냅니다. tick_interval(예: 1.5초)마다 디스코드 메시지의 edit 메서 호출하여 텍스트를 이어 붙입니다.
        - 개발 이유 및 알고리즘 의도: 디스코드 UI의 한계를 극복하고 시각적 몰입감을 주기 위한 장치입니다. 한 번에 수천 자의 텍스트가 덩그러니 나타나는 것은 TRPG의 긴장감을 떨어뜨리므로, 게임 마스터나 화자가 실제로 타이핑을 치고 있는 듯한 연출을 수학적 슬라이싱과 비동기 sleep을 통해 구현했습니다.

3. session.py (세션 초기화 및 환경 프로비저닝 모듈)
    이 모듈은 새로운 TRPG 세션을 시작할 때 필요한 디스코드 서버 내의 물리적 채널 구축과 AI의 컨텍스트(기억) 공간을 할당하는 '초기화 엔진'입니다.

    3.1. create_session (!새세션 [시나리오명])
        - 작동 방법 (Operation):
            * 입력받은 scenario_id를 통해 core.load_scenario_from_file을 호출하여 정적 JSON 데이터를 읽어옵니다.
            * uuid.uuid4()를 사용하여 중복될 확률이 극히 낮은 8자리 고유 식별자(Session ID)를 생성합니다.
            * 해당 식별자를 이름으로 하는 로컬 디렉토리(sessions/[ID])를 생성합니다.
            * 디스코드 API를 호출하여 서버 내에 독립된 카테고리를 만들고, 일반 플레이어용 game- 채널과 GM 전용 master- 채널을 생성합니다. 이때 PermissionOverwrite를 사용하여 master- 채널은 봇과 GM만 읽을 수 있도록 철저히 격리합니다.
            * core.TRPGSession 객체를 인스턴스화하고, build_scenario_cache_text를 호출하여 제미나이 서버에 장기 기억 캐시(Context Cache)를 6시간짜리 TTL(Time-To-Live)로 발급받으며, 생성 시점에는 순수 업로드(입력) 비용만 원화(KRW)로 정산하여 보고합니다.
            * 최종 완성된 세션 객체를 bot.active_sessions 메모리에 올리고 data.json으로 직렬화합니다.
        - 개발 이유 및 알고리즘 의도 (Intent):
            * 완벽한 샌드박스(Sandbox) 구축: TRPG는 몰입이 중요하며, 다른 세션의 로그나 외부 인원의 채팅이 섞이면 게임이 망가집니다. 명령어 한 번으로 UUID 기반의 격리된 채널과 디렉토리를 프로비저닝(Provisioning)함으로써, GM의 환경 세팅 피로도를 0으로 수렴하게 만듭니다.
            * 사전 캐싱을 통한 병목 제거: 게임이 시작된 후 텍스트를 캐싱하면 첫 턴 판정에 심각한 API 지연(Delay)이 발생합니다. 세션 생성 시점에 수만 자의 시나리오 룰북을 구글 서버에 미리 올려둠으로써, 실제 게임 진행 시점에는 캐시 읽기(Cached Read)만 수행하여 응답 속도를 비약적으로 끌어올리기 위한 알고리즘적 안배입니다.

    3.2. start_game (!시작)
        - 작동 방법 (Operation):
            * 현재 채널과 연결된 세션 객체를 불러오고, session.is_started 플래그가 True인지 검사합니다. 이미 참이라면 명령어 실행을 차단합니다.
            * scenario_data의 start_message를 스트리밍 연출로 게임 채널에 출력합니다.
            * 핵심 로직: 해당 시작 메시지를 텍스트 형태로 session.raw_logs 배열에 role="model"의 권한으로 주입(Inject)하고 데이터를 저장합니다.
        - 개발 이유 및 알고리즘 의도 (Intent):
            * 동기화 및 발화자 조작: AI에게 주어지는 프롬프트 배열(raw_logs)의 첫 시작을 시스템이나 유저(role="user")가 아닌, AI 자신(role="model")이 말한 것으로 조작하는 해킹 기법입니다. 이렇게 하면 AI는 "내가 방금 이 세계관의 서막을 묘사했구나"라고 스스로 착각하게 되어, 이후 유저의 행동(role="user")이 입력되었을 때 자연스럽게 게임 마스터의 스탠스를 유지하며 상황을 이어가게 됩니다.
            * 중복 방지 플래그: is_started 플래그는 치명적인 데이터 오염을 막습니다. 실수로 !시작을 두 번 치게 되면 시작 메시지가 raw_logs에 두 번 복사되어 AI의 문맥 파악에 심각한 할루시네이션을 유발할 수 있기 때문입니다.

    3.3. send_intro (!소개)
        - 작동 방법 (Operation): .env에서 불러온 bot.intro_text와 JSON의 scenario_intro, 그리고 캐릭터 생성을 위한 pc_template의 키값들을 하나의 긴 문자열로 병합한 뒤, core.stream_text_to_channel을 통해 문단 단위로 게임 채널에 타이핑하듯 출력합니다.
        - 개발 이유 및 알고리즘 의도 (Intent): 유저 온보딩(Onboarding) 프로세스의 자동화입니다. 시나리오마다 스탯의 종류(근력, 지능, 혹은 민첩 등)가 다르기 때문에, 하드코딩된 안내문이 아니라 동적으로 pc_template의 키(Key)를 추출하여 보여줌으로써 플레이어가 정확히 어떤 포맷으로 !참가 명령어를 준비해야 하는지 헷갈리지 않게 안내합니다.

4. system.py (유지보수 및 리소스 제어 모듈)
    디스코드 서버의 한계와 외부 API 과금 체계를 관리하여, 장기적인 봇 운영을 가능하게 하는 관리자 전용 툴킷입니다.

    4.1. cleanup_channels (!채널정리)
        - 작동 방법 (Operation):
            * ctx.guild 내의 모든 카테고리와 텍스트 채널을 순회하며 이름에 "TRPG", "세션", "game-", "master-"가 포함된 객체를 딕셔너리에 필터링합니다.
            * 필터링된 객체들을 core.ChannelDeleteView라는 UI 클래스에 넘겨, 디스코드 드롭다운 메뉴(Select Menu) 형태로 채팅창에 띄웁니다.
            * 관리자가 UI에서 항목을 선택하고 [영구 삭제]를 누르면, 채널/카테고리 삭제와 동시에 내부적으로 _cleanup_session_memory가 호출되어 bot.active_sessions에 상주하던 해당 세션의 메모리를 강제로 해제(Pop)합니다.
        - 개발 이유 및 알고리즘 의도 (Intent):
            * 가비지 컬렉션(Garbage Collection): 디스코드 서버는 채널 개수에 엄격한 제한(보통 500개)이 있습니다. 테스트용으로 만들어진 수많은 더미 세션 채널들을 일일이 우클릭으로 지우는 것은 매우 고통스러운 작업이므로 이를 자동화했습니다.
            * 메모리 누수(Memory Leak) 방지: 디스코드 채널만 지우고 파이썬 메모리(active_sessions)에 객체를 남겨두면 심각한 메모리 누수가 발생합니다. UI 클래스 내부에서 채널 삭제와 함께 딕셔너리 참조를 끊어 파이썬의 가비지 컬렉터가 객체를 수거할 수 있도록 안전하게 디자인되었습니다.

    4.2. manage_cache (!캐시 [재발급/삭제])
        - 작동 방법 (Operation):
            * 삭제: session.cache_name을 이용해 구글 genai_client.caches.delete API를 호출하여 서버의 캐시를 명시적으로 폭파하고, 객체의 상태를 None으로 갱신합니다. 이 과정에서 캐시가 유지된 보관 시간을 초 단위에서 분 단위로 반올림하여 정확한 스토리지 유지 비용을 정산합니다.
            * 재발급: 기존 캐시가 있다면 먼저 삭제(try-except로 실패 무시) 및 보관 비용 정산을 수행한 후, core.build_scenario_cache_text를 다시 호출하여 새로운 캐시를 생성, 업로드 비용을 과금 추적기(total_cost)에 즉시 누적시킵니다.
        - 개발 이유 및 알고리즘 의도 (Intent): 비용 통제력 확보: Gemini API의 Context Cache는 사용하는 순간만이 아니라 '보관하고 있는 매 시간'마다 스토리지 비용이 과금됩니다. TRPG 세션이 일주일 뒤에 재개된다면, 7일 동안 캐시를 유지하는 것은 엄청난 비용 낭비입니다. 따라서 게임이 끝날 때 GM이 수동으로 !캐시 삭제를 명시하여 과금의 수도꼭지를 잠그고, 다음 주 세션 직전에 !캐시 재발급을 눌러 비용 효율을 극대화하기 위해 개발된 재무적(Financial) 통제 장치입니다.

    4.3. end_session (!세션종료)
        - 작동 방법 (Operation): 게임 채널의 @everyone 권한 중 send_messages를 False로 변경하여 채널을 잠급니다. 이후 session.cache_name이 존재할 경우 제미나이 API에 명시적 삭제를 요청하고 분 단위 보관 비용을 원화(KRW)로 즉시 정산합니다. 마지막으로 게임 채널에는 개입하지 않고 마스터 채널에만 채널 잠금, 캐시 파기, 요금 정산 내역을 통합하여 보고합니다.
        - 개발 이유 및 알고리즘 의도 (Intent): 캐시 유지 시간을 연장(6시간)함에 따라 세션 종료 후 캐시를 방치했을 때 발생하는 불필요한 스토리지 과금을 원천 차단하기 위한 필수 안전장치입니다. 마스터가 세션을 마칠 때 채팅 잠금과 캐시 파기를 개별 명령어로 수행하는 번거로움과 휴먼 에러를 방지하기 위해 하나의 원자적(Atomic) 트랜잭션으로 통합했습니다.

    4.4. reload_cog (!리로드 [모듈명])
        - 작동 방법 (Operation): bot.reload_extension(f"cogs.{cog_name}")을 호출하여, 메모리 상에서 실행 중인 파이썬 확장 모듈을 다시 컴파일하여 덮어씌웁니다.
        - 개발 이유 및 알고리즘 의도 (Intent): 무중단 업데이트(Zero-Downtime Maintenance). 게임을 진행하다가 프롬프트 알고리즘의 오류를 발견하거나 주사위 로직을 긴급하게 수정해야 할 때, 봇 프로세스 자체(main.py)를 재시작하면 BGM 재생이 끊기고 모든 세션의 메모리가 일시적으로 날아가는 불편함이 발생합니다. 이 명령어를 통해 핵심 엔진(main.py, core.py)이 아닌 세부 기능(cogs)의 코드 변경 사항을 런타임에 즉시 핫스왑(Hot-swap)할 수 있도록 구현했습니다.

5. character.py (캐릭터 데이터 파이프라인 모듈)
    이 모듈은 플레이어(PC)와 시스템(NPC)의 데이터를 읽고, 쓰고, 조작하는 모든 행동(CRUD)을 담당합니다. 코어 엔진의 TRPGSession 객체 내부의 딕셔너리에 직접 접근하여 상태를 변이시킵니다.

    5.1. join_session (!참가 [캐릭터명])
        - 작동 방법:
            * 명령어 입력 채널이 game_ch_id인지 확인하여 마스터가 실수로 참가하는 것을 방지합니다.
            * 시나리오 데이터에 정의된 빈 스탯 구조체인 pc_template을 .copy() 메서드를 통해 복제합니다.
            * session.players 딕셔너리에 사용자의 디스코드 고유 ID(문자열)를 키(Key)로 하여 생성된 프로필 딕셔너리를 할당합니다.
            * 권한 허용 시, 디스코드 사용자의 서버 닉네임을 입력받은 char_name으로 즉시 변경합니다.
        - 개발 이유 및 알고리즘 의도:
            * 참조 얕은 복사 방지: 파이썬 딕셔너리 특성상 .copy() 없이 템플릿을 그대로 할당하면 모든 플레이어가 동일한 메모리 주소를 공유하게 되어, 한 명의 체력이 깎이면 모두의 체력이 깎이는 치명적 버그가 발생합니다. 이를 방지하기 위한 데이터 독립성 확보 로직입니다.
            * UI/UX 통합: 닉네임 강제 변경은 플레이어들의 디스코드 메타데이터를 통일시켜 게임 몰입감을 높이고, 이후 game.py의 자동 채팅 로깅 시스템이 닉네임을 정확히 파싱할 수 있게 돕는 전처리 과정입니다.

    5.2. set_profile (!설정 [캐릭터명] [항목] [내용]) 및 adjust_stat (!증감 [캐릭터명] [항목] [수치])
        - 작동 방법:
            * 두 명령어 모두 core.get_uid_by_char_name을 이용해 문자열 이름으로 디스코드 고유 ID를 역추적합니다.
            * 변경하려는 key가 원본 pc_template에 존재하는지 유효성(Validation) 검사를 수행합니다.
            * !설정은 단순 문자열 덮어쓰기를 수행합니다.
            * !증감은 딕셔너리의 기존 값(Value)과 입력받은 amount를 각각 int()로 캐스팅하여 수학적 덧셈을 수행한 뒤, 다시 문자열로 캐스팅하여 저장합니다. 변동량은 +d 포맷팅을 통해 기호(+/-)를 포함하여 게임 채널에 방송(Broadcast)됩니다.
        - 개발 이유 및 알고리즘 의도:
            * 무결성 강제: 존재하지 않는 항목(예: '체력' 대신 오타로 '채력')을 추가하려 할 경우, 딕셔너리 키가 오염되어 AI의 프롬프트가 망가집니다. 이를 막기 위해 엄격한 화이트리스트(Whitelist) 검증을 거칩니다.
            * 연산 편의성 제공: TRPG 시스템 특성상 자원(HP, MP, 골드 등)의 소모와 획득이 빈번합니다. 마스터가 암산하여 !설정 아서 체력 15라고 치는 대신 !증감 아서 체력 -5로 직관적인 상태 제어를 할 수 있도록 분기된 편의 기능입니다. 값은 JSON 직렬화의 안정성을 위해 항상 문자열 타입으로 변환되어 저장됩니다.

    5.3. manage_appearance (!외형) 및 show_profile (!프로필)
        - 작동 방법:
            * !외형은 매개변수 appearance의 전달 여부에 따라 Getter(조회)와 Setter(저장) 역할을 동시에 수행하는 오버로딩(Overloading) 패턴을 따릅니다.
            * !프로필은 디스코드의 시각적 컴포넌트인 discord.Embed 객체를 인스턴스화합니다. 플레이어의 프로필 아바타 URL을 긁어와 썸네일로 박고, 딕셔너리를 순회(items())하며 모든 스탯을 인라인 필드(inline=True)로 배치하여 게임 채널에 송출합니다.
        - 개발 이유 및 알고리즘 의도:
            * 데이터 분리: 능력치(숫자)와 외모 묘사(긴 문자열)를 분리 관리합니다. AI가 주사위 판정을 할 때는 숫자만 참조하고, 전투 묘사를 할 때는 외모를 함께 참조하게 하는 등 프롬프트 최적화의 기반이 됩니다.
            * 데이터 가시화: 복잡한 JSON 내부 데이터를 플레이어가 읽기 편한 GUI 형태(Embed)로 렌더링하는 뷰(View) 역할입니다.

    5.4. manage_npc (!엔피씨 [행동] [이름] [내용])
        - 작동 방법: action 인자에 따라 딕셔너리 연산을 분기합니다 (설정/확인/삭제/목록). session.npcs 공간을 조작합니다.
        - 개발 이유 및 알고리즘 의도: 시나리오 JSON에 정의된 default_npcs는 정적 데이터입니다. 게임이 진행되며 특정 NPC가 부상을 입거나 태도가 변했을 때, AI가 이 변경점을 인지하게 하려면 캐시된 룰북이 아닌 동적 메모리를 수정해야 합니다. 이 명령어는 PromptBuilder가 우선적으로 참조하는 session.npcs를 조작하여, 정적 시나리오 설정을 동적으로 오버라이드(Override)하는 우선순위 제어 시스템입니다.

    5.5. generate_character_cmd (!설정생성 [pc/npc] [이름] [지시사항])
        - 작동 방법:
            * 명령어 입력을 받아 API 호출을 대기 상태로 넘기고(Blocking 방지), 안내 메시지를 먼저 출력합니다.
            * core.generate_character_details를 비동기로 호출합니다. 이때, 빠른 응답 속도 중심의 DEFAULT_MODEL 대신 논리 추론력이 높은 LOGIC_MODEL을 사용합니다.
            * 반환된 객체의 usage_metadata를 뜯어 토큰 소모량을 계산하고 session.total_cost에 합산합니다.
            * 결과물 문자열의 길이가 디스코드 전송 제한(2000자)을 넘을 경우, 리스트 슬라이싱 기법([i:i+2000])을 활용하여 메시지를 분할 전송합니다.
        - 개발 이유 및 알고리즘 의도:
            * 창작 부하 경감 및 일관성 확보: 즉석에서 NPC가 등장해야 할 때 마스터가 세계관에 맞는 설정을 전부 짜는 것은 큰 부담입니다. AI를 '보조 작가'로 활용하여, 시나리오의 톤 앤 매너(Tone & Manner)를 위반하지 않는 설정을 도출해 내기 위한 로직입니다.
            * 정교한 과금 추적기 연동: 프로 모델(LOGIC_MODEL)은 기본 모델에 비해 단가가 훨씬 비쌉니다. 따라서 이 함수가 호출될 때마다 토큰량을 정밀하게 추출하여 로그와 메모리에 즉시 과금을 누적시켜 재무적 통제를 잃지 않도록 합니다.

6. media.py (시청각 몰입 엔진 모듈)
    텍스트 위주의 TRPG 환경에 이미지 에셋과 BGM/효과음을 비동기적으로 제공하여 감각적 몰입도를 높이는 서브시스템입니다.

    6.1. send_media (!이미지 [키워드/목록])
        - 작동 방법:
            * 명령어 인자가 '목록'일 경우, session.scenario_data의 media_keywords 딕셔너리 키값들을 추출해 Embed로 출력합니다.
            * 키워드일 경우, 딕셔너리에서 실제 파일명을 매핑(Mapping)한 뒤, 파이썬 내장 라이브러리 os.path.join을 사용하여 경로를 안전하게 조립(media/[시나리오ID]/[파일명])합니다. 이후 discord.File 객체로 패키징하여 게임 채널에 업로드합니다.
        - 개발 이유 및 알고리즘 의도: 경로 보호 및 휴먼 에러 방지. 사용자가 !이미지 ./../config.json 등과 같이 상위 디렉토리로 접근하는 해킹 공격(Path Traversal)을 시도하거나, 복잡한 확장자명(.jpeg, .png 등)을 입력하다 오타를 내는 것을 원천 차단합니다. 마스터는 오직 시나리오 작성 시 정의한 '한글 키워드'만으로 안전하고 빠르게 에셋을 꺼낼 수 있습니다.

    6.2. play_bgm (!브금 [파일명/목록/정지])
        - 작동 방법:
            * 사용자가 접속해 있는 음성 채널 상태를 추적(ctx.author.voice.channel)하여 봇을 해당 채널에 연결(connect())하거나 이동(move_to())시킵니다.
            * 재생: discord.FFmpegPCMAudio를 이용해 오디오 소스를 읽고, PCMVolumeTransformer로 볼륨 제어 래퍼(Wrapper)를 씌웁니다.
            * 루핑 알고리즘: vc.play(source, after=after_playing) 구조를 사용합니다. 음악이 끝났을 때 트리거되는 after 콜백 함수 내부에 다시 동일한 음악을 재생하는 재귀적 호출 로직을 넣어 메인 스레드 블로킹 없이 무한 반복을 구현합니다.
            * 정지 및 교체 연출: 음악을 즉각 멈추면(vc.stop()) 위화감이 크므로, asyncio.sleep과 volume * 0.8을 결합한 fade_out 비동기 태스크를 백그라운드에 던져, 소리가 2~3초에 걸쳐 부드럽게 줄어들며 잦아들게(Fade-out) 하는 오디오 후처리 알고리즘을 사용합니다.
        - 개발 이유 및 알고리즘 의도: 일반적인 디스코드 음악 봇과 달리 TRPG 전용이므로 '반복 재생(Loop)'과 '분위기 전환(Fade-out)'이 가장 중요합니다. 스트리밍 링크(YouTube 등)가 아닌 로컬 media/ 경로의 오프라인 파일을 사용하여 지연 없는 즉각적 재생과 안정성을 담보합니다.

    6.3. playlist_control (!플리 [행동] (시나리오명))
        - 작동 방법:
            * 특정 세션에 종속되지 않고 서버(Guild) 단위로 작동합니다. core.PlaylistManager 클래스를 인스턴스화합니다.
            * media_dir 내의 모든 mp3 파일을 긁어와 리스트(Queue)로 만든 후 random.shuffle()로 순서를 섞습니다.
            * 비동기 큐 제어: manager.player_loop()라는 독립된 백그라운드 코루틴(Task)이 실행됩니다. 곡이 하나 끝날 때마다 asyncio.Event().set()을 트리거하여 다음 곡이 재생되게 하며, '다음/이전' 명령어가 들어오면 큐의 인덱스 방향(skip_direction)을 변경하여 트랙을 제어합니다.
        - 개발 이유 및 알고리즘 의도: BGM 하나만 루프시키는 것을 넘어, 환경음(예: 빗소리, 여관 소음 등 여러 트랙)이나 음악 폴더 자체를 라디오처럼 백그라운드에 깔아두는 수요를 만족시키기 위한 독립 모듈입니다. 메인 TRPG 봇 로직과 스레드를 철저히 분리하여, 플레이리스트 연산이 주사위 판정이나 AI 텍스트 생성 속도에 전혀 영향을 미치지 않도록 설계되었습니다.

    6.4. control_chat (!채팅 [상태])
        - 작동 방법: discord.TextChannel.set_permissions를 이용해 채널의 기본 권한인 @everyone 역할의 send_messages 옵션을 Boolean(True/False) 값으로 토글합니다.
        - 개발 이유 및 알고리즘 의도: AI나 마스터가 장문의 묘사를 스트리밍하고 있을 때, 플레이어가 채팅을 치면 화면이 섞여 몰입도가 깨지거나 스트리밍 코드가 충돌할 우려가 있습니다. 이를 제어하기 위해 묘사 시작 전 채팅을 막고 끝나면 풀어주는 '하드웨어적 잠금 장치'입니다.

7. game.py (메인 게임 엔진 모듈)
    이 모듈은 단순한 명령어 처리를 넘어, 자연어 파싱, 비동기 통신, 상태 관리, 비용 최적화가 가장 복잡하게 얽혀 있는 시스템의 심장부입니다.

    7.1. 이벤트 리스너: on_message (자동 로깅 시스템)
        - 작동 방법 (Operation):
            * 디스코드 채널에 메시지가 올라올 때마다 트리거됩니다. (봇 자신의 메시지나 !로 시작하는 명령어는 무시합니다.)
            * 메시지가 올라온 채널 ID를 검사하여 master_ch_id인지 game_ch_id인지 판별합니다.
            * 마스터 채널일 경우: GM의 일반 채팅을 게임 채널로 릴레이(Relay) 스트리밍하고, session.current_turn_logs에 [진행자]: 내용 형태로 저장합니다.
            * 게임 채널일 경우: session.players를 검색해 디스코드 닉네임 대신 설정된 캐릭터명으로 치환하여 [캐릭터명]: 내용 형태로 로그를 저장합니다.
            * 각각의 채팅 내용은 core.write_log를 통해 로컬 텍스트 파일(master_chat_log.txt, game_chat_log.txt)에도 영구 기록됩니다.
        - 개발 이유 및 알고리즘 의도 (Intent):
            * 투명한 데이터 수집: TRPG는 말과 행동이 모두 데이터가 됩니다. 매번 명령어를 쳐서 행동을 선언하는 것은 플레이어의 피로도를 높이므로, 일반 채팅을 치기만 해도 자동으로 누가 어떤 행동을 했는지 식별하여 프롬프트의 재료(current_turn_logs)로 쓸 수 있게 만드는 백그라운드 수집기입니다.
            * 채널 격리 및 릴레이: 마스터는 게임 채널에 직접 나타나지 않고 마스터 채널에서만 타자를 쳐도, 시스템이 그 말을 게임 채널로 옮겨줍니다. 이는 마스터를 '신'과 같은 존재로 연출하여 게임의 신비감을 유지하려는 의도입니다.

    7.2. 명령어: request_dice (!주사위)
        - 작동 방법 (Operation):
            * param1이 숫자인지 문자열인지 판별하여 일반 판정과 능력치 판정으로 분기합니다.
            * 일반 판정: 1~N면체 주사위를 굴립니다. 목표값(target_val)이 주어지면 성공/실패 여부를 판독합니다.
            * 능력치 판정: session.players에서 해당 캐릭터의 특정 스탯 수치를 불러와 이를 목표값으로 삼아 판정합니다.
            * 양쪽 모두 core.GeneralDiceView 또는 core.DiceView 객체를 게임 채널에 출력하여 플레이어가 직접 버튼을 누르게 만듭니다. 버튼을 누른 결과값과 성공 여부는 session.current_turn_logs에 강제 기록됩니다.
        - 개발 이유 및 알고리즘 의도 (Intent):
            * 행위의 직렬화(Serialization): 주사위를 굴렸다는 사실과 그 수치는 AI가 다음 상황의 개연성을 판정하는 가장 중요한 근거입니다. 버튼 클릭 한 번으로 무작위성이 결과로 고정되고, 이것이 즉시 AI의 프롬프트 배열로 편입되도록 알고리즘적으로 완벽히 연결해 두었습니다.

    7.3. 핵심 턴 진행 엔진: proceed_turn (!진행)
        전체 시스템에서 가장 중요한 코루틴(Coroutine)으로, 인간의 지시와 AI의 묘사를 연결하는 파이프라인입니다.

        7.3.1. 전처리 알고리즘 (Tag Parsing & State Mutation)
            - 작동 방법: 정규표현식(re)을 사용하여 GM이 입력한 instruction에서 특수 문법 태그를 추출하고 데이터를 조작합니다.
                * 상/중/하:[이미지키워드] -> 출력할 이미지 목록 큐에 담기
                * 자:[캐릭터];[아이템];[수치] -> session.resources 딕셔너리의 아이템 개수 연산
                * 태:[캐릭터];[상태] -> session.statuses 리스트에서 상태이상 추가(+) 또는 제거(-)
            - 개발 이유: LLM에게 "아서에게 독을 걸고 열쇠를 줘라"라고 프롬프트로 지시해도, LLM이 시스템 내부의 인벤토리 데이터까지 완벽하게 수정해주지는 못합니다. 따라서 프롬프트를 전송하기 '직전'에 정규식으로 태그를 가로채어 파이썬 딕셔너리의 상태(State)를 확정적으로 갱신한 뒤, 태그가 지워진 깨끗한 텍스트만 AI에게 넘겨 할루시네이션 없는 무결한 데이터 제어를 이뤄냅니다.

        7.3.2. 캐시 자가 복구 알고리즘 (Retry Logic)
            - 작동 방법: 내부 함수 generate_with_retry(retry_count)를 정의하여 재귀적으로 호출합니다.
                * AI에게 GenerateContent 요청을 보냈을 때 APIError(특히 400, 404 캐시 만료 에러)가 발생하면, 예외 처리 블록(except APIError)이 이를 낚아챕니다.
                * retry_count가 0일 경우, 유저에게 알림을 띄우고 만료된 이전 캐시의 보관 비용을 정산한 뒤, 그 자리에서 즉시 core.build_scenario_cache_text를 실행해 새 캐시를 구글 서버에 결제 및 발급받습니다.
                * 발급이 완료되면, retry_count=1 상태로 자신을 다시 호출하여 중단되었던 턴 묘사를 완벽히 이어갑니다.
            - 개발 이유: Gemini 캐시는 6시간 동안 호출되지 않으면 소멸합니다. 게임 중 밥을 먹고 오거나 화장실을 다녀온 뒤 !진행을 쳤을 때 캐시가 증발하여 세션이 통째로 뻗어버리는 치명적 장애를 방지하는 무중단 백업(Fallback) 시스템입니다.

        7.3.3. 묘사 스트리밍 및 이미지 동기화
            - 작동 방법: AI의 응답 텍스트를 문단(\n\n)으로 쪼갭니다.
                * 상 태그 이미지는 첫 번째 문단을 칠 때, 하 태그 이미지는 모든 텍스트가 끝난 후 전송됩니다.
                * 중 태그 이미지는, 스트리밍 중인 텍스트 내에 이미지 키워드(예: 크리쳐_묘사)가 등장하는 시점을 텍스트 탐색으로 파악하여 해당 타이밍에 정확히 삽입합니다.
            - 개발 이유: 긴 텍스트 묘사 중간에 상황에 맞는 적절한 삽화가 튀어나오도록 연출하여, 텍스트 기반 TRPG의 단조로움을 깨고 시각적 박진감을 극대화합니다.

        7.3.4. 백그라운드 기억 압축 릴레이 (Background Compression)
            - 작동 방법: 한 턴의 묘사가 무사히 종료되고 session.turn_count를 1 올린 뒤, 만약 현재 턴이 5의 배수(5턴, 10턴...)라면 기억 압축 로직을 비동기적으로 스케줄링합니다.
                * session.uncompressed_logs에 모아둔 5턴 치의 텍스트를 core.build_compression_prompt에 넣어 로직 모델(Pro)에게 보냅니다.
                * 반환된 건조한 개조식 요약을 session.compressed_memory에 덧붙이고(Append), 압축이 완료된 원본 로그 리스트는 메모리에서 삭제(del)합니다.
            - 개발 이유: 이 시스템의 진정한 핵심입니다. 일반적인 챗봇은 대화가 길어지면 앞에 했던 말을 통째로 잊거나 토큰 제한에 걸려 터집니다. 이 시스템은 5턴마다 '무손실 압축'을 수행하여 장기 기억 공간을 최적화함으로써, 세션이 수백 턴(몇 달) 동안 이어져도 AI가 1턴에 일어났던 일과 소지품 상태를 절대 잊지 않고 일관성을 유지하도록 만듭니다.

    7.4. 수동 제어 명령어
        7.4.1. compress_memory (!기억압축)
            - 작동 방법: !진행 내부에 들어있는 자동 압축 알고리즘과 로직은 100% 동일하지만, 5턴을 채우지 않았어도 강제로 압축을 실행시킵니다.
            - 개발 이유: 세션을 종료하고 내일 다시 이어서 하려 할 때, 잔여 로그를 깔끔하게 압축하여 상태를 저장해두고 싶을 때 사용하는 GM의 수동 제어 트리거입니다.

        7.4.2. manage_note (!노트 [누적/갱신/출력] (내용))
            - 작동 방법: session.note 문자열을 조작합니다. 이 문자열은 core.PromptBuilder.add_note_block을 통해 매 턴 프롬프트의 최상단에 주입됩니다.
            - 개발 이유: AI의 압축 메모리마저 신뢰할 수 없는 절대적인 규칙(예: "현재 방 안의 중력은 거꾸로 작용하고 있다")이나 반드시 명심해야 할 GM의 속마음을 시스템 메모리 영역에 하드코딩(Hardcoding) 수준으로 박아넣기 위한 강제 제어 장치입니다.

        7.4.3. manage_cache_note (!캐시노트 [누적/갱신/출력] (내용))
            - 작동 방법: session.cache_note 문자열을 조작합니다. 이 문자열은 즉각 프롬프트에 주입되지 않으며, 차후 캐시가 재발급될 때 코어 룰북 텍스트의 마지막에 지연 병합(Lazy Merge)되어 함께 구워집니다.
            - 개발 이유: 세계관이나 빈 시나리오의 핵심 설정이 대규모로 실시간 추가될 때 매번 프롬프트 텍스트로 넘기면 토큰 비용이 막대하게 발생합니다. API 통신 없이 내부 메모리만을 갱신해 두었다가, 세션 단위로 한 번만 캐시에 구워 스토리지 비용으로 전환하기 위한 과금 최적화 우회 기법입니다.
